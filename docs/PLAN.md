# CircuitProxy — PLAN

## Видение

**CircuitProxy** — учебный L7 reverse proxy с встроенными resilience-паттернами,
написанный на Go с нуля (не обёртка над Traefik/nginx). Цель — глубоко разобраться
в паттернах устойчивости распределённых систем: health checks, circuit breaker
(closed/open/half-open), retry с backoff, graceful degradation.

Это **не бизнес-идея** — экономика не оценивается. Мотивация: искренняя техническая
сложность и обучение. Бюджет: $0 (только stdlib + опционально стандартные Go-инструменты).

## Почему это не тривиальная обёртка

Главная техническая мина — **корректный half-open state circuit breaker'а**.
Когда circuit переходит из `open` в `half-open`, нужно пропустить **ровно один**
пробный запрос, а не лавину одновременных запросов. Под параллельной нагрузкой
легко случайно пропустить сразу несколько пробных запросов — это источник тонких
конкурентных багов, которые не видны в однопоточном тесте, но проявляются под
реальной нагрузкой. Именно эта конкурентная корректность — сердце проекта.

## Верхнеуровневая архитектура

```
                     ┌──────────────────────────────────────────┐
   client ──HTTP──▶  │              CircuitProxy                 │
                     │                                            │
                     │  ┌────────────┐   ┌──────────────────┐     │
                     │  │  Balancer  │──▶│  Backend pool     │     │
                     │  │ round-robin│   │  (per-backend     │     │
                     │  └────────────┘   │   circuit breaker)│     │
                     │        ▲          └────────┬─────────┘     │
                     │        │                   │               │
                     │  ┌─────┴──────┐   ┌────────▼─────────┐     │
                     │  │ Healthcheck│   │ ReverseProxy      │─────┼──▶ backend N
                     │  │  (active   │   │ (httputil +       │     │
                     │  │   ping)    │   │  retry/backoff)   │     │
                     │  └────────────┘   └──────────────────┘     │
                     └──────────────────────────────────────────┘
```

Ключевые компоненты:

- **`internal/proxy`** — reverse proxy поверх `net/http/httputil.ReverseProxy`,
  round-robin выбор backend'а, интеграция с breaker'ом и retry.
- **`internal/breaker`** — конечный автомат circuit breaker (closed/open/half-open)
  с конкурентно-корректным half-open через atomic CAS.
- **`internal/healthcheck`** — активный периодический ping backend'ов, исключение
  недоступных из ротации.
- **`internal/config`** — парсинг конфиг-файла (список backend'ов, пороги breaker'а).
- **`cmd/circuitproxy`** — CLI-точка входа.

## Список Этапов (MVP)

| Этап | Название | Суть |
|------|----------|------|
| 0 | Скелет | `go mod`, пакеты-заглушки, CLI-каркас, конфиг-парсинг |
| 1 | Базовый reverse proxy | `httputil.ReverseProxy` + round-robin по статическому списку backend'ов |
| 2 | Активный health-check | периодический ping, исключение недоступных из ротации |
| 3 | Circuit breaker | 3 состояния + конкурентно-корректный half-open (ровно один пробный запрос) |
| 4 | Retry с backoff | ретрай при неудаче с учётом идемпотентности HTTP-метода |
| 5 | Конфиг + наблюдаемость | конфиг-файл, метрики и структурное логирование состояний breaker'а |

Каждый Этап — отдельная ветка от `master` → PR → merge. Подробности и критерии
готовности в [TECHNICAL_PLAN.md](TECHNICAL_PLAN.md).

## После MVP

Nice-to-have (не входит в MVP, вырезано осознанно):

- Weighted balancing (веса backend'ов).
- Sticky sessions (привязка клиента к backend'у).
- TLS termination.
- Множественные backend-группы + маршрутизация по пути/хосту.

Подробнее — [POST_MVP_PLAN.md](POST_MVP_PLAN.md).

## Что вырезано из v1 (и почему)

- **Weighted balancing / sticky sessions / TLS** — размывают фокус. v1 — только
  plain HTTP round-robin + circuit breaker.
- **Кластеризация / распределённое состояние breaker'а** — breaker живёт в памяти
  одного процесса; это осознанно.
- **Динамическая перезагрузка конфига** — конфиг читается один раз при старте.
