# CircuitProxy — TECHNICAL_PLAN

Детальный технический план: стек, архитектура, конкурентная модель и разбивка по Этапам.

## Стек

- **Язык:** Go 1.23+ (toolchain `~/sdk/go/bin`).
- **Зависимости:** только stdlib. Reverse proxy — `net/http/httputil.ReverseProxy`;
  конкурентность — `sync/atomic`, `sync`; HTTP-сервер — `net/http`; логирование —
  `log/slog` (stdlib, структурное). Никаких внешних модулей в MVP.
- **Тесты:** стандартный `testing` + `net/http/httptest`. Гонки ловим `go test -race`.
- **Конфиг:** JSON-файл (stdlib `encoding/json`) — без внешних парсеров YAML/TOML,
  чтобы держать нулевые зависимости.

## Решение по Docker (явно)

**Docker / docker-compose НЕ используется** — ни для продукта, ни для тестов.

Обоснование: для проверки circuit breaker'а нужны тестовые backend'ы, которые можно
временно «уронить» и вернуть. Это полностью и удобнее покрывается `httptest.Server`
внутри самих Go-тестов: сервер поднимается in-process, его handler можно на лету
переключать на возврат 500/таймаута (имитация падения) и обратно (имитация
восстановления) через `atomic.Bool`, а `Close()` имитирует полную недоступность.
Никакой сетевой оркестрации контейнеров не требуется, тесты остаются
детерминированными и быстрыми (`go test -race ./...`). Поэтому Docker в проекте
отсутствует. Если позже понадобится ручной end-to-end прогон против «настоящих»
процессов — достаточно `go run` пары мини-серверов из `examples/`, а не Compose.

## Структура пакетов

```
circuitproxy/
├── cmd/circuitproxy/main.go   # CLI: флаги, загрузка конфига, запуск сервера
├── internal/
│   ├── config/                # парсинг и валидация конфига (JSON)
│   ├── proxy/                  # ReverseProxy + round-robin balancer + retry-обвязка
│   ├── breaker/               # circuit breaker state machine (atomic)
│   └── healthcheck/           # активный периодический health-check
├── docs/
└── go.mod
```

`internal/` — вся логика непубличная (это приложение, не библиотека). Слои:
`config` → зависит ни от кого; `breaker` → ни от кого; `healthcheck` → от `breaker`
(помечает backend up/down); `proxy` → от `breaker` + `config`; `cmd` → собирает всё.

## Конкурентная модель circuit breaker (ядро проекта)

Три состояния на **каждый backend**:

- **closed** — запросы идут нормально. Считаем подряд идущие ошибки; при достижении
  `FailureThreshold` → переход в `open`.
- **open** — все запросы к backend'у немедленно fast-fail (`ErrBreakerOpen`), без
  реального обращения. По истечении `OpenTimeout` → переход в `half-open`.
- **half-open** — пропускается **ровно один** пробный запрос. Успех → `closed`
  (сброс счётчиков); неудача → снова `open` (таймаут перезапускается).

### Как гарантируется «ровно один» пробный запрос

Состояние хранится в `atomic.Int32` (кодирует state) + отдельный
`trialInFlight atomic.Bool`. Алгоритм на входе (`Allow() (bool, ...)`):

1. Читаем текущее состояние атомарно.
2. Если `closed` → разрешаем.
3. Если `open` → проверяем, истёк ли `OpenTimeout` (сравнение с `atomic` временем
   перехода). Если не истёк → fast-fail. Если истёк → пытаемся атомарно перевести
   `open → half-open` через **CAS** на поле состояния. Победитель CAS продолжает в
   ветку half-open; проигравшие CAS видят, что состояние уже `half-open`, и идут в п.4.
4. Если `half-open` → пытаемся `trialInFlight.CompareAndSwap(false, true)`. **Ровно
   один** вызов выиграет CAS и получит право на пробный запрос; все остальные
   параллельные вызовы получают `false` (fast-fail, стратегия ниже).

Результат пробного запроса (`Report(success bool)`) сбрасывает `trialInFlight` и
переводит состояние: успех → `closed`, неудача → `open`. CAS на двух независимых
atomic-полях (`state` для перехода open→half-open, `trialInFlight` для допуска
единственного пробника) исключает гонку «лавина пробников».

### Стратегия для НЕ-пробных запросов во время half-open

**Выбрано: fast-fail** (не блокирующее ожидание). Запросы, не выигравшие
`trialInFlight`-CAS, немедленно получают `ErrBreakerOpen`, как если бы circuit был
`open`. Обоснование:

- Простота и отсутствие блокировок/очередей горутин (нет риска накопления
  ожидающих под нагрузкой).
- Half-open — короткое переходное окно; отправлять клиента ждать результата
  единственного пробника хуже, чем сразу вернуть быструю ошибку (клиент/вышестоящий
  ретрай сам решит, что делать). Ожидание рассматривается как post-MVP улучшение.

Это задокументированный контракт: **во время half-open ровно один запрос пробует
backend, остальные fast-fail.**

## Разбивка по Этапам

### Этап 0 — Скелет (этот bootstrap)

- `go mod init github.com/akomyagin/circuitproxy`.
- Пакеты-заглушки `internal/proxy`, `internal/breaker`, `internal/healthcheck`,
  `internal/config` с `// TODO(Этап N)`.
- `cmd/circuitproxy/main.go` — парсинг флага `-config`, чтение JSON-конфига, вывод
  распарсенного конфига (пока без запуска сервера).
- **Готово когда:** `go build ./...` и `go vet ./...` чисты.

### Этап 1 — Базовый reverse proxy + round-robin

- `internal/proxy`: `Balancer` со статическим списком backend URL'ов, round-robin
  выбор (`atomic.Uint64` счётчик по модулю длины пула).
- Обёртка над `httputil.ReverseProxy`: на каждый входящий запрос выбираем backend,
  проставляем `Director`/`Rewrite`, проксируем.
- HTTP-сервер в `cmd`, поднимается на адресе из конфига.
- **Тесты:** несколько `httptest.Server` как backend'ы; проверяем, что запросы
  распределяются round-robin; проверяем корректность проксирования тела/заголовков.
- **Готово когда:** запросы равномерно раскладываются по backend'ам, `-race` чист.

### Этап 2 — Активный health-check

- `internal/healthcheck`: горутина с `time.Ticker`, периодически шлёт GET на
  health-путь каждого backend'а (`Interval`, `Timeout`, `HealthPath` из конфига).
- Backend помечается `up`/`down` (atomic-флаг на backend). Balancer пропускает
  `down`-backend'ы при round-robin выборе.
- Graceful: health-checker останавливается по `context.Context` при завершении.
- **Тесты:** `httptest.Server` с переключаемым health-handler'ом (`atomic.Bool`);
  проверяем, что «упавший» backend исключается из ротации и возвращается после
  восстановления.
- **Готово когда:** ротация корректно реагирует на смену состояния backend'ов.

### Этап 3 — Circuit breaker (ядро)

- `internal/breaker`: реализация автомата по разделу «Конкурентная модель» выше.
- Публичный API: `New(cfg) *Breaker`, `Allow() (bool, error)`, `Report(success bool)`.
- Интеграция в `proxy`: перед проксированием — `Allow()`; после — `Report()`
  (успех/неуспех определяется по статус-коду и ошибке транспорта).
- **Тесты (критично):**
  - Однопоточные: closed→open по порогу, open→half-open по таймауту, half-open→closed
    и half-open→open.
  - **Конкурентные (`-race`):** N горутин параллельно вызывают `Allow()` в момент
    перехода open→half-open; ассертим, что **ровно один** получил `true` (пробник).
  - Stress: множество горутин крутят Allow/Report в цикле — детектор гонок молчит.
- **Готово когда:** «ровно один пробник» доказан конкурентным тестом под `-race`.

### Этап 4 — Retry с backoff

- В `proxy`: при неудачном запросе к backend'у (ошибка транспорта / 5xx, если
  политика включает) — повтор на **другом** backend'е (или том же после backoff),
  до `MaxRetries`, с экспоненциальным backoff (`base * 2^n` + jitter).
- **Идемпотентность:** ретраятся только идемпотентные методы (GET, HEAD, PUT,
  DELETE, OPTIONS, TRACE). POST/PATCH по умолчанию **не** ретраятся (если тело уже
  прочитано / метод не идемпотентен). Тело буферизуется при необходимости повтора
  идемпотентного запроса с телом.
- Ретрай уважает circuit breaker: не повторяем на backend с `open` circuit.
- **Тесты:** backend, отдающий N ошибок затем успех — проверяем повтор и итоговый
  успех; POST не ретраится; backoff-задержки применяются (инжектируемый clock/sleep).
- **Готово когда:** идемпотентные ретраятся, не-идемпотентные — нет, backoff растёт.

### Этап 5 — Конфиг + наблюдаемость

- `internal/config`: финальная схема JSON (listen addr, backends[], health-check
  params, breaker thresholds, retry policy) + валидация с понятными ошибками.
- Наблюдаемость: структурное логирование (`log/slog`) переходов состояний breaker'а
  (closed→open→half-open→closed) с backend-меткой; счётчики (запросы, ошибки,
  fast-fail, ретраи) — простой in-memory `expvar` или `/metrics`-эндпоинт с
  текстовым выводом (без внешнего Prometheus-клиента в MVP).
- **Тесты:** парсинг/валидация конфига (table-driven, включая невалидные конфиги);
  логирование переходов проверяется через `slog` с buffer-handler'ом.
- **Готово когда:** конфиг валидируется, переходы breaker'а видны в логах/метриках.

## Тестовая стратегия (сквозная)

Согласно портфельной конвенции — **пирамида с интеграционным ярусом**, а не только
мок-HTTP юнит-тесты:

- **Unit:** чистая логика (round-robin индекс, backoff-расчёт, конфиг-валидация).
- **Integration (детерминированные фейки):** реальный `http.Server` проксирует на
  реальные `httptest.Server`-backend'ы; падения/восстановления имитируются
  переключаемыми handler'ами. Гоняет реальный сетевой стек in-process.
- **Concurrency (`-race`):** конкурентные тесты half-open и stress-тесты breaker'а.

Все тесты обязаны проходить под `go test -race ./...`.
